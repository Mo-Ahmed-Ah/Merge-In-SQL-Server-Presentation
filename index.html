What is MERGE

One Statement.
Three Operations.

The MERGE statement in Microsoft SQL Server allows you to synchronize two tables using a single SQL statement — comparing a source table against a target table and applying all necessary changes atomically.

＋

INSERT New Rows

When a row exists in the Source but not in the Target, MERGE automatically inserts it.

⟳

UPDATE Existing Rows

When a matching row is found in both tables, MERGE updates the Target with Source values.

✕

DELETE Stale Rows

When a row exists in Target but is no longer in Source, MERGE deletes it.

💡

Introduced in SQL Server 2008

MERGE is ANSI SQL:2003 compliant and has been available since SQL Server 2008. It's ideal for ETL pipelines, staging-to-production syncs, and upsert patterns.

Setting Up the
Employee Tables.

We work with two tables: Empolyes (the target/production table) and UpdateEmpolyes (the source/staging table).

01_create_tables.sql 
-- ① Create the TARGET table (production)
CREATE TABLE Empolyes (
    NationalId     INT          NOT NULL,
    EmpName        VARCHAR(100) NOT NULL,
    EmpPhoneNumber VARCHAR(20)  NOT NULL,
    EmpSalary      DECIMAL(10,2) NOT NULL

-- ② Create the SOURCE table (staging / updated data)
CREATE TABLE UpdateEmpolyes (
    NationalId     INT          NOT NULL,
    EmpName        VARCHAR(100) NOT NULL,
    EmpPhoneNumber VARCHAR(20)  NOT NULL,
    EmpSalary      DECIMAL(10,2) NOT NULL
);

The Problem Without MERGE

Multiple Queries,
One Problem.

To synchronize employee data without MERGE, you'd need three separate queries — increasing complexity, error surface, and maintenance burden.

WITHOUT MERGE — 3 queries

  • Separate UPDATE statement for existing employees
  • Separate INSERT statement for new employees
  • Separate DELETE statement for removed employees
  • Multiple database round-trips
  • Complex manual transaction handling
  • Race condition risk between statements
  • Hard to read & maintain
VS

WITH MERGE — 1 statement

  • Single atomic MERGE statement
  • All operations in one database pass
  • Built-in transactional integrity
  • Clear, readable WHEN clauses
  • No race conditions between operations
  • Easy to audit & debug
  • Less code, fewer bugs
without_merge.sql — ❌ Old approach 
-- ❌ Step 1: Update existing employees
UPDATE Empolyes
SET    EmpName        = u.EmpName,
       EmpPhoneNumber = u.EmpPhoneNumber,
       EmpSalary      = u.EmpSalary
FROM   Empolyes e
INNER JOIN UpdateEmpolyes u ON e.NationalId = u.NationalId;

-- ❌ Step 2: Insert new employees
INSERT INTO Empolyes (NationalId, EmpName, EmpPhoneNumber, EmpSalary)
SELECT u.NationalId, u.EmpName, u.EmpPhoneNumber, u.EmpSalary
FROM   UpdateEmpolyes u
WHERE NOT EXISTS (
    SELECT 1 FROM Empolyes e WHERE e.NationalId = u.NationalId
);

-- ❌ Step 3: Delete removed employees
DELETE e
FROM   Empolyes e
WHERE NOT EXISTS (
    SELECT 1 FROM UpdateEmpolyes u WHERE u.NationalId = e.NationalId
);

MERGE Statement in Action

The Full
MERGE Query.

merge_employees.sql — ✅ Single statement 
MERGE Empolyes AS Target
USING UpdateEmpolyes AS Source
  ON  Target.NationalId = Source.NationalId

-- Case 1: Row exists in BOTH tables → UPDATE
WHEN MATCHED THEN
    UPDATE SET
        Target.EmpName        = Source.EmpName,
        Target.EmpPhoneNumber = Source.EmpPhoneNumber,
        Target.EmpSalary      = Source.EmpSalary

-- Case 2: Row in Source only → INSERT
WHEN NOT MATCHED BY TARGET THEN
    INSERT (NationalId, EmpName, EmpPhoneNumber, EmpSalary)
    VALUES (
        Source.NationalId,
        Source.EmpName,
        Source.EmpPhoneNumber,
        Source.EmpSalary
    )

-- Case 3: Row in Target only → DELETE
WHEN NOT MATCHED BY SOURCE THEN
    DELETE;   -- semicolon is REQUIRED to end MERGE
🎯

MERGE … USING … ON

Declares the target (Empolyes), the source (UpdateEmpolyes), and the join key (NationalId) used to match rows.

⟳

WHEN MATCHED → UPDATE

Fires when NationalId exists in both tables. Updates phone number and salary in the production table.

＋

WHEN NOT MATCHED BY TARGET → INSERT

Fires when NationalId is in Source but missing from Target. Asma Ahmed (ID 4) is a new hire — MERGE inserts her.

✕

WHEN NOT MATCHED BY SOURCE → DELETE

Fires when NationalId is in Target but missing from Source. Mohamed Ahmed (ID 3) left the company — MERGE removes him.

Advantages of MERGE

Why MERGE
Wins.

Beyond just being shorter code, MERGE brings structural and architectural advantages to your data synchronization strategy.

01

Single Atomic Statement

All INSERT, UPDATE, and DELETE operations execute in a single atomic transaction. Either all changes commit — or none do. No partial sync states.

02

Reduced Code Complexity

Replace 3 separate DML statements and a complex transaction wrapper with one clean, expressive MERGE statement that fits in a single screen.

03

Improved Readability

The WHEN clauses read like business rules: "When matched → update. When new → insert. When gone → delete." Self-documenting by design.

04

Simplifies Synchronization

Ideal for ETL pipelines, data warehouse loads, and any upsert pattern. One statement replaces complex staging logic.

05

Performance Gains

A single pass through the data means fewer I/O operations, fewer lock acquisitions, and less network overhead compared to three separate queries.

Best Practices

Use MERGE
Correctly.

MERGE is powerful, but requires careful setup to behave predictably and perform well at scale.

🔑

Use a Unique Matching Column

Always match on a primary key or unique identifier — like NationalId in our example. If the ON condition can match multiple rows in Source, MERGE will throw an error or cause unpredictable updates.

🧪

Test with Sample Data First

Run MERGE against a small representative subset in a development environment. Verify row counts for each WHEN clause match your expectations before touching production.

⚡

Index the Join Columns

Ensure NationalId is indexed on both the source and target tables. Without indexes, MERGE will perform full scans and degrade significantly with large datasets.

🛡️

Verify Before Production

Wrap MERGE in a BEGIN TRANSACTION … ROLLBACK during testing to inspect results before committing. Use the OUTPUT clause to capture what each row operation did.

safe_test_pattern.sql 
-- ✅ Safe testing pattern with transaction
BEGIN TRANSACTION;

MERGE Empolyes AS Target
USING UpdateEmpolyes AS Source
  ON  Target.NationalId = Source.NationalId
WHEN MATCHED THEN
    UPDATE SET
        Target.EmpPhoneNumber = Source.EmpPhoneNumber,
        Target.EmpSalary      = Source.EmpSalary
WHEN NOT MATCHED BY TARGET THEN
    INSERT (NationalId, EmpName, EmpPhoneNumber, EmpSalary)
    VALUES (Source.NationalId, Source.EmpName,
            Source.EmpPhoneNumber, Source.EmpSalary)
WHEN NOT MATCHED BY SOURCE THEN
    DELETE;

-- Inspect before committing
SELECT * FROM Empolyes;

-- Rollback to undo — or COMMIT to save
ROLLBACK;   -- change to COMMIT when ready
⚠️

Important Limitation

Each source row must match at most one target row. Duplicate NationalId values in the Source table will cause MERGE to fail or behave unexpectedly. Always ensure your join key is unique in the source.
